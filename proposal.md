# 提案说明书

## 课题名称

用 AI Agent + 工作流引擎构建企业级智能运维控制台，让运维工程师通过对话完成日常巡检、故障诊断和自动化操作，释放运维团队 80% 的重复性工作量

---

## 一句话价值主张

为基础架构部门和运维团队，解决「巡检慢、诊断难、自动化门槛高」三大痛点——通过 AI Agent + 可视化工作流引擎，将自然语言对话转化为跨系统的自动化运维动作。**运维侧**：日常巡检从人工逐项排查变为 AI 一键执行，平均耗时从 2 小时降至 5 分钟；**业务侧**：故障定位从多工具切换变为单一控制台对话诊断，MTTR（平均修复时间）降低 70%。

---

## 第一章：问题定义与洞察

### 1.1 背景与现状

企业基础架构团队面临规模化运维的核心矛盾：服务数量持续增长，但运维工程师规模无法同步扩张。以中等规模企业为例，基础架构团队通常管理 50-200 个服务实例，涵盖中间件（Kafka、RabbitMQ、Elasticsearch）、存储（PostgreSQL、MinIO）、监控（Prometheus、Grafana）、DevOps 工具链（Artifactory、GitLab、Harbor）等多个类别。

**当前运维模式的核心问题：**

运维工程师日常工作高度碎片化——登录不同系统查看健康状态、手动执行巡检脚本、在多个工具间切换排查故障。这种"人肉运维"模式不仅效率低下，还高度依赖个人经验，知识无法沉淀和复用。

尽管市场上存在各类运维工具，但它们普遍面临同一个困境：**工具孤岛**。Prometheus 负责监控告警，Ansible 负责自动化执行，各系统有独立的 Dashboard，运维工程师需要在多套系统之间手动串联操作，缺乏统一的智能化入口。

AI 大模型的成熟为打破这一困局提供了可能——通过 AI Agent 理解自然语言意图并调用工具链，实现「说一句话，完成一套操作」的智能运维体验。

- **涉及角色**：基础架构运维工程师（平台建设与日常运维）、SRE（稳定性保障）、业务开发（服务依赖方，需要了解服务状态）
- **现有流程**：接到告警 → 登录多个系统逐一排查 → 手动执行修复脚本 → 更新工单记录 → 人工撰写巡检报告。全流程高度依赖人工操作，每次故障响应平均需要 3-4 名工程师协同
- **知识现状**：运维操作经验积累在个人脑中，无法系统化沉淀为可复用的工作流和技能库

| 关键指标 | 当前值 | 数据来源 |
|---|---|---|
| 单次日常巡检耗时 | 1.5-2 小时/人（逐系统登录检查） | 工程师排期估算 |
| 故障定位平均耗时（MTTR 诊断阶段） | 30-90 分钟（多工具切换排查） | 工单记录统计 |
| 运维经验复用率 | 极低（依赖口耳相传和 Wiki 文档） | 团队评估 |
| 工作流自动化覆盖率 | <20%（主要靠脚本，缺乏统一编排） | 当前自动化任务数估算 |

### 1.2 痛点/机会分析

#### 效率提升类

- **目标用户？** 基础架构运维团队（3-8 人，负责全公司服务运维）、SRE 工程师（负责稳定性保障）、业务开发（需要了解依赖服务的健康状态）
- **效率卡点？** 运维工作存在三个关键卡点：
  1. **工具分散**：每个服务有独立管控界面，运维工程师需要在 5-10 个系统间切换，认知负担高
  2. **经验孤岛**：故障诊断依赖个人经验，同类问题重复排查，无法沉淀为可复用的自动化操作
  3. **自动化门槛高**：Ansible/Shell 脚本编写需要专业能力，业务运维人员无法自助创建工作流
- **使用情况？** 假设团队管理 100 个服务，每日巡检 2 小时，每月发生 20-30 次告警响应，每次平均 45 分钟。每月仅重复性运维工作消耗约 80-120 人时

**用户故事：**

> 「作为基础架构团队的运维工程师，我每天需要登录十几个不同的管理后台逐一检查服务状态，然后把结果汇总成巡检报告。一旦出现告警，还要在多个系统间来回切换排查。我希望有一个统一的智能控制台，能用自然语言描述我的需求，让 AI 帮我执行巡检、诊断故障、生成报告——我只需要审批关键操作，而不是亲自执行每一个步骤。」

### 1.3 量化价值预期

#### 效率提升类

| 价值维度 | 指标 | Before | After | 提升幅度 | 推算依据 |
|---|---|---|---|---|---|
| 效率 | 日常巡检耗时 | 1.5-2 小时/次（人工逐系统检查） | 5-10 分钟/次（AI Agent 自动执行巡检工作流） | ~90% | 基于 Demo 工作流执行测试 |
| 效率 | 故障定位耗时（MTTR 诊断阶段） | 30-90 分钟（多工具切换排查） | 5-15 分钟（AI 对话式诊断 + 工具调用） | ~70% | 基于 AI Agent 工具调用链路测试 |
| 成本 | 运维重复性工作量（月） | 约 80-120 人时/月 | <20 人时/月（工作流自动化覆盖率 >80%） | >75% | 基于工单量和单次耗时推算 |
| 质量 | 运维操作规范化程度 | 依赖个人习惯，无统一标准 | 工作流即规范，每步操作有记录和结果追踪 | 从无到有 | 工作流执行日志全量留存 |
| 知识 | 运维经验沉淀 | 零散在个人脑中和 Wiki | Skills 技能库统一管理，可被 AI Agent 直接调用 | 从人到平台 | 技能库条目数量持续增长 |

> **推算过程：**
>
> - **巡检效率**：传统巡检 = 登录各系统（10 min）× N 个系统 + 逐项检查（60 min）+ 汇总报告（30 min）≈ 1.5-2h。AI 工作流巡检 = 一句话触发 → Agent 并行调用各系统检查接口 → 自动生成报告 ≈ 5-10 min。
> - **故障定位**：传统路径 = 告警接收 → 登录 Prometheus 查指标 → 登录日志系统查错误 → 手动执行诊断命令 → 多人协同分析 ≈ 30-90 min。AI 路径 = 自然语言描述故障现象 → Agent 自动执行诊断工具链 → 结构化输出根因分析 ≈ 5-15 min。
> - **工作量节省**：假设每月巡检 20 次（2h/次）+ 告警响应 25 次（45min/次）= 約 59 人时。平台化后巡检降至 10min/次 + 响应降至 15min/次 = 约 10 人时，节省约 83%。

## 第二章：解决方案

### 2.1 方案概述

Hi-Ops 是一个基于 AI Agent + 工作流引擎的企业级智能运维控制台。作为基础架构团队的统一运维入口，它通过三层架构实现智能化运维：**对话层**（AI Agent 理解自然语言并执行工具调用）、**编排层**（YAML 工作流定义可复用的自动化操作序列）、**资产层**（统一管理服务资产、技能库和执行历史）。运维工程师通过自然语言对话完成日常巡检、故障诊断和批量操作，AI 辅助贯穿产品开发和运维执行全流程。

---

### 2.2 核心功能

| 功能模块 | 功能描述 | 业务价值 |
|---|---|---|
| AI 对话运维 | 多模型支持的 AI Agent 对话界面，支持流式输出、思维链展示、工具调用可视化和人工审批（HITL）工作流 | 运维工程师用自然语言完成操作，零学习成本；关键操作有人工审批保障安全 |
| 可视化工作流引擎 | YAML 定义的工作流，支持步骤结果追踪、执行历史记录、成功率统计和图表分析；支持定时触发和手动执行 | 将运维经验编码为可复用工作流，执行过程全程可视化、可审计 |
| 技能库（Skills Marketplace） | 插件化 AI 技能管理，支持技能注册、分类浏览、调用追踪；技能可被 AI Agent 动态调用 | 运维知识资产化，技能可跨场景复用；新技能上线无需修改主流程 |
| 服务资产中心 | 统一管理 100+ 公共服务，YAML 数据模型支持集群状态、集成状态、工作流绑定、巡检 Checklist | 一个界面掌握所有服务状态，告别多系统切换 |
| 执行历史与统计 | 工作流运行记录、步骤结果卡片、成功率趋势图、执行时间分析 | 运维操作全量留存，支持合规审计和问题回溯 |
| Keycloak SSO 集成 | 统一身份认证，Token 生命周期管理，会话过期自动重定向 | 企业级安全接入，无需维护独立账号体系 |

### 2.3 用户旅程

**Before（现状）：**

```
步骤 1：接收告警通知（角色：运维工程师，耗时：即时）
  ↓
步骤 2：登录 Prometheus 查看指标（角色：运维工程师，耗时：5-10 min）
  ↓
步骤 3：登录日志系统（ELK/Loki）查看错误日志（角色：运维工程师，耗时：10-15 min）
  ↓
步骤 4：SSH 到服务器执行诊断命令（角色：运维工程师，耗时：10-20 min）
  ↓
步骤 5：呼叫相关人员协同排查（角色：多人协同，耗时：15-30 min）
  ↓
步骤 6：执行修复操作，更新工单（角色：运维工程师，耗时：10-20 min）
  ↓
结果：故障恢复（总耗时：50-95 分钟，消耗 2-3 名工程师）
```

**After（使用 Hi-Ops 平台后）：**

```
步骤 1：在 AI Chat 描述故障现象（角色：运维工程师，耗时：30 秒）  ← 自然语言输入
  ↓
步骤 2：AI Agent 自动调用诊断工具链（Prometheus 查询/日志分析/健康检查）（角色：AI，耗时：1-2 min）  ← 自动执行
  ↓
步骤 3：AI 输出结构化根因分析和修复建议（角色：AI，耗时：30 秒）  ← 思维链可视化
  ↓
步骤 4：运维工程师审批关键修复操作（HITL）（角色：运维工程师，耗时：1 min）  ← 人工确认
  ↓
步骤 5：平台执行修复，自动记录操作日志（角色：平台，耗时：2-3 min）  ← 自动完成
  ↓
结果：故障恢复（总耗时：5-7 分钟，1 名工程师主导）
```

### 2.5 提案亮点

| 创新点 | 常见做法 | 我们的做法 | 为什么更好 |
|---|---|---|---|
| AI Agent 工具调用可视化 | AI 回复纯文本，工具调用过程黑盒 | 每个工具调用实时展示调用状态（running/done/error）+ 结构化入参/出参展示 + 思维链（Chain of Thought）透明呈现 | 运维工程师能理解 AI 的决策过程，建立信任；问题可追溯 |
| HITL 人工审批集成 | 自动化操作要么全自动（风险高）要么全人工（效率低） | 关键操作（重启服务、变更配置）触发 HITL 审批节点，AI 暂停执行等待工程师确认，审批通过后继续 | 兼顾效率与安全：80% 的只读诊断操作全自动，20% 的变更操作有人工把关 |
| 技能即插件架构 | 运维自动化能力固化在平台代码中，扩展需要改代码发版 | Skills 技能库：每个技能是独立的 YAML + 脚本定义，可独立注册、启用/禁用；AI Agent 运行时动态发现可用技能 | 运维知识资产化，新技能上线无需改平台代码；各团队可自主贡献领域技能 |
| 工作流结果追踪 | 工作流只记录成功/失败，步骤细节不可见 | 每个步骤输出独立结果卡片，支持 Markdown 渲染、代码高亮、数据图表；汇总统计成功率趋势 | 运维操作过程全透明，从执行到审计形成完整闭环 |
| AI 辅助平台建设 | 运维平台需要 3-5 人团队数月开发 | 单人 + Claude Code AI 编程助手，完成 Next.js 前端（React 19 + Tailwind v4）+ AI Elements 组件库 + SSE 流式引擎 + 工作流引擎全栈开发 | 验证了 AI 辅助开发复杂前端系统的可行性，大幅降低平台建设门槛 |
