### 课题名称

用 AI + Kubernetes 构建企业级数据库自助服务平台，让业务开发 2 分钟获得带 REST API 的完整数据库服务，释放 DBA 90% 重复交付工作量

### 一句话价值主张

为基础架构部门和 DBA 团队，解决 PostgreSQL 数据库对外提供服务时「交付慢、门槛高、管理难」三大痛点——通过 Kubernetes + Supabase 开源组件自动编排，让业务线自助申请即获得带 REST API、认证、管理界面的完整数据库服务。核心价值：**业务侧**从申请到使用仅需 2 分钟（原需 3-4 天），且零 CRUD 代码量；**DBA 侧**从逐个手动交付变为平台化审批监控，重复性工作减少 90% 以上。

## 第一章：问题定义与洞察

> **本章目标：** 让评委相信你发现了一个真实、重要、值得解决的问题。
>
> **核心原则：** 用数据和事实说话，不要拍脑袋。

### 1.1 背景与现状

PostgreSQL 是企业最主流的关系型数据库，但「给业务提供数据库服务」这件事一直是痛点：DBA 手动建库配权限，业务开发拿到裸连接后还得自己写一套 CRUD API——重复、低效、且每个团队各写各的。

Supabase（GitHub 75,000+ Stars）给出了一个优雅的思路：在 PostgreSQL 之上自动生成 REST API（PostgREST）+ 认证（GoTrue）+ 可视化管理界面，让数据库直接变成开箱即用的后端服务。这意味着**业务开发不用再写 CRUD 接口，DBA 不用再逐个手动交付**。

然而，DBA 团队在将 PostgreSQL 数据库对外提供服务时，面临大量重复性的运维工作：每个业务线申请数据库，DBA 需要手动创建实例、配置权限、提供连接信息，业务开发还需要自己写 CRUD 接口。Supabase 虽然能解决单个项目的问题，但其官方自托管方案是**单项目架构**，无法满足企业内多个业务线同时使用的场景——要服务 10 个业务线就得部署 10 套独立的 Supabase，运维复杂度线性爆炸。

- **涉及角色**：基础架构运维开发（平台建设）、DBA（数据库交付与管理）、业务开发（使用数据库服务）
- **现有流程**：业务线提交数据库申请 → DBA 手动创建 PostgreSQL 实例 → 配置用户权限 → 业务开发自行编写 CRUD 接口 → 联调测试。全流程需要 DBA 和业务开发多次沟通协调
- **安全现状**：如果采用 Supabase 自托管，全局共享 JWT Secret，多个业务线之间无法实现 token 隔离

| 关键指标 | 当前值 | 数据来源 |
|---|---|---|
| DBA 交付一个数据库服务的周期 | 1-2 天（含沟通、创建、配置、联调） | DBA 工单记录 |
| 业务开发为数据库编写 CRUD 接口耗时 | 2-3 天 | 开发排期估算 |
| JWT 密钥隔离 | 无（Supabase 自托管全局共享） | 官方自托管架构文档 |

### 1.2 痛点/机会分析

#### 效率提升类

- **目标用户？** DBA 团队（3-5 人，负责全公司数据库交付与运维）、业务开发团队（10+ 个业务线，每个线 3-10 名开发）
- **效率卡点？** 整条链路存在两个关键卡点：
  1. **DBA 交付环节**：每次数据库申请都需要 DBA 手动操作（建库、配权限、给连接串），DBA 成为业务上线的瓶颈
  2. **业务开发环节**：拿到裸数据库连接后，每个业务线都要重复编写一套 CRUD API 接口，工作雷同但无法复用
- **使用情况？** 假设每月 5-8 个业务线提出数据库服务申请，每次 DBA 交付耗时半天到一天，业务开发再花 2-3 天写 CRUD 接口。每月仅此一项，DBA 侧消耗 3-8 人天，业务开发侧消耗 10-24 人天

**用户故事：**

> 「作为基础架构部门的运维开发，我的职责是帮助 DBA 团队把 PostgreSQL 数据库更高效地对外提供服务。目前每当业务线需要一个新的数据库服务，DBA 要手动创建实例、配置用户权限、提供连接信息，业务开发再基于裸连接自己写 CRUD 接口——整个流程 DBA 要花半天到一天，业务开发还要再花两三天写接口层。我希望做一个平台，业务线自助申请就能拿到一个带 REST API 和管理界面的完整数据库服务，DBA 只需在平台上审批和监控，不用再逐个手动交付。」

## 第二章：解决方案

> **本章目标：** 让评委清楚你怎么解决问题，以及 AI 在其中扮演什么角色。
>
> **核心原则：** 说清楚「AI 做了什么」而非「用了什么工具」。

### 2.1 方案概述

Superbase 是一个基于 Kubernetes 的多租户数据库服务平台（DBaaS）。作为基础架构部门为 DBA 和业务开发打造的内部平台，它通过 Go 控制面 + Helm Chart 自动编排，将 PostgreSQL 从裸数据库升级为带 REST API、认证和管理界面的完整服务——业务线自助创建项目即可获得开箱即用的数据库服务，DBA 从手动交付转变为平台化管理。AI 辅助贯穿开发全流程，Claude Code 作为 AI 编程助手参与了架构设计、代码实现和安全测试。

---

### 2.2 核心功能

| 功能模块 | 功能描述 | 业务价值 |
|---|---|---|
| 自助式项目创建 | 业务开发在 Dashboard 一键创建项目，平台自动完成 K8s Namespace、PostgreSQL、PostgREST、认证服务的全套部署 | 业务无需等待 DBA，从申请到可用仅需 2 分钟 |
| 自动 REST API 生成 | 基于 PostgREST，PostgreSQL 表自动映射为标准 REST API，支持 CRUD、过滤、排序、分页，兼容 Supabase SDK | 业务开发零 CRUD 代码量，建表即可用 API |
| 可视化数据库管理 | Dashboard 提供表编辑器、SQL 控制台、Schema 可视化、RLS 策略管理等完整工具 | DBA 和业务开发均可通过界面操作，降低使用门槛 |
| 多业务线安全隔离 | Per-Project JWT 密钥 + K8s Namespace 物理隔离 + NetworkPolicy 网络隔离 | 各业务线数据完全隔离，平台自动保证安全，消除人工误配风险 |
| 统一监控与管理 | 控制面统一管理所有项目的状态、资源、密钥，集成 Prometheus 监控 | DBA 从逐个运维转为平台化管控，一个界面管所有项目 |

### 2.3 用户旅程

Before vs After，对比展示：没有本方案时用户怎么做，有了之后怎么做。

**Before（现状）：**

```
步骤 1：业务线提交数据库申请工单（角色：业务开发，耗时：10 min）
  ↓
步骤 2：DBA 排队处理，创建 PostgreSQL 实例并配置权限（角色：DBA，耗时：半天~1天）
  ↓
步骤 3：DBA 提供连接信息，与业务开发沟通表结构和权限需求（角色：DBA+业务开发，耗时：1-2h）
  ↓
步骤 4：业务开发基于裸数据库连接编写 CRUD API 接口（角色：业务开发，耗时：2-3 天）
  ↓
步骤 5：联调测试，修复权限和接口问题（角色：DBA+业务开发，耗时：半天）
  ↓
结果：数据库服务可用（总耗时：3-4 天）
```

**After（使用 Superbase 平台后）：**

```
步骤 1：业务开发在 Dashboard 点击「创建项目」（角色：业务开发，耗时：10 秒）  ← 自助操作
  ↓
步骤 2：平台自动创建 PostgreSQL + PostgREST + 认证服务（角色：平台，耗时：90 秒）  ← 自动完成
  ↓
步骤 3：业务开发在 Dashboard 建表，通过 SDK 直接调用 REST API（角色：业务开发，耗时：30 min）  ← 无需写 CRUD
  ↓
结果：数据库服务可用（总耗时：约 30 分钟，DBA 零介入）
```

```
                    │  Go Controller │  (项目 CRUD API / Helm 编排)

                            │
              ┌─────────────┼─────────────┐
              │             │             │
      ┌───────┴──┐   ┌─────┴─────┐  ┌────┴─────┐
      │ Keycloak │   │ 控制面 DB  │  │Prometheus│
      │  (认证)  │   │(项目元数据)│  │  (监控)  │
      └──────────┘   └───────────┘  └──────────┘
                  superbase-system namespace
    ─────────────────────────────────────────────────
                  project-{id} namespace (× N)
              ┌─────────────┬─────────────┐
              │             │             │
      ┌───────┴──┐   ┌─────┴─────┐  ┌────┴─────┐
      │PostgreSQL│   │ PostgREST │  │  GoTrue  │
      │ (独立DB) │   │ (REST API)│  │  (认证)  │
      └──────────┘   └───────────┘  └──────────┘
                            │
                    Nginx Ingress (子域名路由)
                    {project-id}.api.example.com
```

**可落地性说明：**

- **已验证**：在 Kind 本地 K8s 集群上完成全流程验证，包括项目创建、SDK 集成测试、安全隔离测试
- **技术选型成熟**：Kubernetes + Helm + Nginx Ingress 均为生产级基础设施组件
- **渐进式扩展**：Helm Chart 已预留 Supavisor（连接池）、Realtime（实时通信）、Storage（对象存储）、Edge Functions（边缘函数）的模板，可按需启用
